# 01 · 前後端分工

> **這份文件的目的：** 讓前後端不會互踢皮球。每個功能落在哪一邊都先寫清楚。
> **作者：** 前端工程師（合約方提案，後端同仁可以提異議修訂）
> **更新時機：** 任何新功能加入時，先在這裡標歸屬。

---

## 心法：誰擁有「真相之源」誰負責

| 資料的真相在哪 | 由誰負責 |
|---|---|
| **資料庫裡** | 後端（讀寫 + 驗證 + 業務邏輯） |
| **瀏覽器裡（重整就消失）** | 前端 |
| **localStorage / Cookie** | 前端（但敏感資料不能存這） |
| **Token / Session** | 後端發、前端收著傳 |

## 灰色地帶：常見爭議點

### 1. 資料驗證
- **前端做**：即時 UX 提示（例：email 格式錯誤馬上紅字）
- **後端做**：最終把關（即使前端被繞過，後端必須擋下）
- **規則：兩邊都要做，後端為準**

### 2. 商品價格 / 折扣計算
- **前端做**：顯示用，根據 API 給的 `price` + `discount` 算給用戶看
- **後端做**：結帳時權威重算（防止前端被竄改價格）
- **規則：前端算的價只用來顯示，下單時後端忽略前端算的金額**

### 3. 圖片
- **前端做**：CSS 排版、`<img>` 標籤、響應式 `srcset`
- **後端做**：存放（路徑 / S3）、產不同尺寸（thumbnail / medium / full）
- **規則：API 回傳一個物件 `{ url, thumb, alt }`，前端用 `url` 或 `thumb`**

### 4. 表單錯誤訊息
- **前端做**：欄位旁的紅字 UI
- **後端做**：422 回應的 `errors` 物件
- **規則：API 回 422 時 errors 是 `{ field: ["訊息"] }`，前端逐欄位顯示**

### 5. 國際化（多語）
- **前端做**：UI 字典檔（zh-TW / en / vi）
- **後端做**：商品名稱 / 文章內容的多語欄位
- **規則：UI 文字前端管，內容文字後端管**
